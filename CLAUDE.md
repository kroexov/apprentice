# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project context

This is a vmkteam `gold-apisrv` template instance. The Go module is `apisrv` (not `apprentice`); the produced binary is also named `apisrv`. The service name lives in `Makefile.mk` (copied from `Makefile.mk.dist` by `make init`) — most `make` targets read `$(NAME)` from there.

Go 1.25. PostgreSQL is required for full tests and `make run`.

## Commands

First-time setup (must run before anything else):

```sh
make init                    # copies Makefile.mk.dist → Makefile.mk and cfg/local.toml.dist → cfg/local.toml
make tools                   # installs mfd-generator, pgmigrator, colgen, golangci-lint
```

Build / run / lint:

```sh
make build                   # builds ./apisrv (CGO disabled, -mod=vendor)
make run                     # go run with cfg/local.toml in -dev mode
make lint                    # golangci-lint v2.8.0 with .golangci.yml
make fmt                     # golangci-lint fmt
make mod                     # go mod tidy + vendor + git add vendor
```

Tests:

```sh
make test                    # full test suite against real DB ($TEST_PGDATABASE)
make test-short              # short tests only — skips DB-backed tests via -test.run="Test[^D][^B]"
make db-test                 # drop+recreate the test DB and load schema (docs/apisrv.sql + docs/init.sql)
go test -run TestName ./pkg/vt   # single test
```

Tests use `goconvey` BDD style and hit a real PostgreSQL — there is no mocking layer. The `Test[^D][^B]` filter in `test-short` is the convention used to gate DB-dependent tests; name new DB-backed tests so the second character is `D` or `B` (e.g. `TestDB...`, `TestBatch...`) to keep them out of `test-short`.

Code generation:

```sh
make generate                # runs `go generate ./pkg/vt` → regenerates vt_zenrpc.go via the zenrpc tool
make type-script-client      # writes ../gold-vt/src/services/api/factory.ts (sibling repo expected)
```

Database / model generation (mfd-generator workflow, all read `docs/model/apisrv.mfd`):

```sh
make db                      # drop+recreate $(PGDATABASE) and load schema
make mfd-xml                 # update mfd from live DB schema
make mfd-model               # regenerate ./pkg/db model files
make mfd-repo NS=<ns>        # generate repos for namespace
make mfd-vt-xml              # update VT-side mfd
make mfd-vt-rpc NS=<ns>      # generate VT RPC scaffolding for a namespace
make mfd-db-test             # generate DB test helpers in ./pkg/db/test
```

`NS` is required for `mfd-repo`, `mfd-vt-rpc`, and `mfd-vt-template`. The Makefile errors out if it is left as the default `"NONE"`.

## Architecture

Entry point `cmd/apisrv/main.go` parses TOML config, initializes the logger (`vmkteam/embedlog`), Sentry, and the go-pg connection, then constructs `app.App`. The `-ts_client` flag short-circuits to print the generated VT TypeScript client and exit.

`pkg/app` is the composition root. `App.Run` registers metrics, echo middleware, debug/pprof, both RPC servers, and metadata, then starts the HTTP listener. It owns the `*echo.Echo` instance, the VT zenrpc server, the go-pg monitor, and the `db.DB` wrapper.

There are **two** zenrpc servers exposed over HTTP, with different auth and audiences:

- `pkg/rpc` — public API, mounted at `/v1/rpc/`. Built by `rpc.New`. SMDBox UI at `/v1/rpc/doc/`, OpenRPC at `/v1/rpc/openrpc.json`, TS client at `/v1/rpc/api.ts`. No auth middleware by default; services are registered in `RegisterAll` in `pkg/rpc/server.go`.
- `pkg/vt` — admin/back-office API ("VT"), mounted at `/v1/vt/`. Built by `vt.New`. Requires the `Authorization2` header — `authMiddleware` in `pkg/vt/middleware.go` resolves the user via `commonRepo.EnabledUserByAuthKey` and stores it in context (use `vt.UserFromContext`). The single exception is `auth.Login`. Updates `LastActivityAt` opportunistically. Services live in `vt_service.go`; namespaces are registered in `pkg/vt/server.go` (`NSAuth`, `NSUser`, …). The TS client at `/v1/vt/api.ts` uses `WithClasses: true` and excludes the VFS namespace.

Adding a new VT RPC service: implement it in `pkg/vt/`, register it in `vt.New`'s `RegisterAll`, then run `make generate` so the zenrpc tool regenerates `vt_zenrpc.go` (which provides the `RPC.<Service>.<Method>` constants used by middleware to bypass auth, etc.). zenrpc reads doc-comment directives like `//zenrpc:login`, `//zenrpc:return`, `//zenrpc:401` to drive generation.

`pkg/db` is the go-pg-based persistence layer. Models, search structs, validators, and repositories are generated by `mfd-generator` from `docs/model/apisrv.mfd` — do not hand-edit the generated files; edit the mfd and re-run the appropriate `make mfd-*` target. `db.DB` wraps `*pg.DB` and adds `RunInLock` (uses `pg_advisory_xact_lock` keyed by CRC64 of a name) and `runInTransaction`. `buildQuery` is the shared query builder used by repos, applying `Filter`s, an optional `Searcher`, the `Pager`, and variadic `OpFunc`s (sorts, joins, etc.).

`go.mod` uses a `replace` directive pointing `github.com/go-pg/pg/v10` at `github.com/vmkteam/pg/v10` — this is intentional (the fork has custom patches). Don't drop it during dependency updates.

VFS (file storage) is optional, gated by `Server.EnableVFS` in config. When enabled, `App.RegisterVFS` wires `vmkteam/vfs` handlers and a separate VFS namespace excluded from the public TS client.

## HTTP surface

| Path | Purpose |
| --- | --- |
| `/v1/rpc/` | Public zenrpc endpoint |
| `/v1/rpc/doc/` | SMDBox UI for public API |
| `/v1/rpc/openrpc.json` | OpenRPC document |
| `/v1/rpc/api.ts` | TypeScript client for public API |
| `/v1/vt/` | VT zenrpc endpoint (requires `Authorization2`) |
| `/v1/vt/doc/` | SMDBox UI for VT |
| `/v1/vt/api.ts` | TypeScript client for VT (with classes) |
| `/status` | Healthcheck (pings DB) |
| `/metrics` | Prometheus metrics |
| `/debug/pprof/*` | pprof |
| `/debug/metadata` | appkit service metadata |
| `/` | Route listing (only when `Server.IsDevel`) |

Default port is 8075 (see `cfg/local.toml.dist` and `deployments/Dockerfile`).

## Conventions worth knowing

- Debug-only middleware (timing, SQL log) is gated by the query string `__level=5` — see `allowDebugFn` in both server packages.
- Errors returned from RPC methods should be wrapped via the package-local helpers (`InternalError`, `ErrUnauthorized`, `ErrNotFound`, …) so zenrpc sets the right HTTP-style code.
- Logging is structured via `embedlog`; `slog.SetDefault` is set in `main`. Prefer `a.Print` / `a.Error` (embedded `embedlog.Logger`) over the global `slog`.
- Config values come from a TOML file; flags are also pulled from env via `namsral/flag` with prefix `APISRV_` (uppercased app name).
- `pkg/db/apprentice_ext.go` holds hand-written extensions to the mfd-generated `ApprenticeRepo` (helpers like `LatestStageScoreByCandidate`, `NextStageAfter`, `IsUniqueViolation`). Add new repo helpers there — never edit `apprentice.go` (regenerated by `make mfd-repo NS=apprentice`).
- Soft-deleted rows (`statusId = 3`) are filtered out by `StatusFilter`. The DB-level uniqueness on `candidates.handle`, `stages.alias`, `stages.order` is partial (`WHERE statusId <> 3`), so deleting a row releases its handle/alias/order for re-use — both the app-level pre-check and the DB constraint agree.

## Done-with-changes checklist

Before declaring any code change complete (especially anything in `pkg/`), run from the repo root:

```sh
make fmt lint test
```

`make fmt` formats via golangci-lint, `make lint` runs the v2.8.0 linter, `make test` exercises the full suite against a real PostgreSQL (`$TEST_PGDATABASE`). All three must be green. Use `make db-test` first if the test DB is empty or schema-stale.
